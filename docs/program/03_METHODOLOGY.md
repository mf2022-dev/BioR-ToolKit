## 1. PURPOSE

This document defines the **methodology** for developing and managing the National Biodefense Program (BioR). It establishes a systematic, evidence-based approach by integrating five internationally recognized frameworks into one coherent method.

This methodology answers: **"How do we go from an idea to a permanent national institution, step by step, using proven methods?"**

---

## 2. METHODOLOGY ARCHITECTURE

The BioR Program uses a **5-layer methodology stack**. Each layer serves a specific purpose. Together, they cover every aspect of program development — from strategic logic to day-to-day project delivery.

```
┌─────────────────────────────────────────────────────────────┐
│  LAYER 1: LOGICAL FRAMEWORK (LogFrame / LFA)                │
│  Purpose: Program logic, cause-and-effect, measurement      │
│  Source:  USAID (1969), World Bank, UN, EU standard          │
├─────────────────────────────────────────────────────────────┤
│  LAYER 2: MSP — MANAGING SUCCESSFUL PROGRAMMES              │
│  Purpose: Program structure, governance, delivery phases     │
│  Source:  UK Cabinet Office / AXELOS, 5th Edition (2020)     │
├─────────────────────────────────────────────────────────────┤
│  LAYER 3: WHO JEE / NAPHS CAPABILITY FRAMEWORK              │
│  Purpose: Define what capabilities to build (scope)          │
│  Source:  WHO International Health Regulations (2005)         │
├─────────────────────────────────────────────────────────────┤
│  LAYER 4: US NATIONAL BIODEFENSE STRATEGY                   │
│  Purpose: Reference model — what "done" looks like           │
│  Source:  White House / NSC (2018, updated 2022)             │
├─────────────────────────────────────────────────────────────┤
│  LAYER 5: PMI / AGILE HYBRID                                │
│  Purpose: Individual project execution and tracking          │
│  Source:  PMI PgMP 4th Ed. (2017) + Agile/Scrum             │
└─────────────────────────────────────────────────────────────┘
```

### How the Layers Work Together

| Question | Layer | Framework | Answers |
|:---|:---|:---|:---|
| "What is the logical structure of the program?" | 1 | LogFrame | Goal → Purpose → Outputs → Activities, with indicators and assumptions |
| "How do we structure and govern the program?" | 2 | MSP | Vision → Blueprint → Tranches → Reviews, with governance and benefits tracking |

## 3. LAYER 1 — LOGICAL FRAMEWORK (LogFrame)

### 3.1 What is LogFrame?

The Logical Framework Approach (LFA) is the **universal standard** for designing, monitoring, and evaluating programs. It forces clear cause-and-effect thinking: if we do these activities, we produce these outputs, which achieve this purpose, which contributes to this goal.

### 3.2 LogFrame Structure

| Level | Description | Indicator | Verification | Assumptions |
|:---|:---|:---|:---|:---|
| **Goal** | The high-level strategic impact | How we measure the impact | Where the evidence comes from | External conditions that must hold |
| **Purpose** | The program's direct outcome | How we measure the outcome | Where the evidence comes from | What must be true for outputs to achieve purpose |
| **Outputs** | The deliverables the program produces | How we measure each output | Where the evidence comes from | What must be true for activities to produce outputs |
| **Activities** | The work done to produce outputs | Resources required | Budget / timeline | What must be true for activities to proceed |

### 3.3 BioR Program LogFrame

| Level | Narrative | Objectively Verifiable Indicators (OVIs) | Means of Verification (MOVs) | Assumptions |
|:---|:---|:---|:---|:---|
| **Goal** | Saudi Arabia has sovereign, integrated national biodefense capability that protects the population from biological threats — natural, accidental, and deliberate | WHO JEE score improvement (target: 4+ across all 19 areas within 10 years); GHS Index ranking improvement; Zero undetected biological events exceeding 72 hours | WHO JEE re-evaluation report; GHS Index annual update; National surveillance system logs | Political stability maintained; Sustained government commitment; International cooperation continues |
| **Purpose** | An integrated national biodefense system is operational, connecting detection, characterization, assessment, and response across all sectors | Number of sectors integrated (target: 6 — health, lab, defense, agriculture, environment, regulatory); Mean time from detection to national alert (target: < 24 hours); National biological threat assessment published (target: quarterly) | System operational reports; Integration test results; Published threat assessments | Participating ministries cooperate; Technical infrastructure is available; Trained workforce is recruited |
| **Output 1** | Unified national biosurveillance detection network | Number of surveillance data sources connected (target: 20+); Percentage of Kingdom territory covered (target: 100%); Real-time data latency (target: < 1 hour) | Platform dashboard; Network status reports; Independent technical audit | Existing surveillance systems provide data access; Network infrastructure exists in all regions |
| **Output 2** | National laboratory information management system (LIMS) | Number of laboratories connected (target: all BSL-2+ facilities); Sample tracking from collection to characterization (target: 100% chain of custody); Genomic sequencing turnaround (target: < 48 hours) | LIMS reports; Laboratory accreditation records; Sample audit trail | Laboratories adopt standardized protocols; Equipment and reagents are procured; ISO 15189/35001 compliance is achieved |
| **Output 3** | Continuous biological threat assessment capability | Number of threat assessments produced (target: 52/year — weekly); Risk scoring covers all 5 domains (biosurveillance, biosecurity, biodefense, biosafety, CBRN); Geospatial intelligence layers active (target: 10+) | Published threat reports; Analytical platform logs; GeoIntel system records | Threat intelligence sources are accessible; Analytical staff are trained; Assessment methodology is validated |
| **Output 4** | Coordinated multi-sector biodefense response mechanism | Number of sectors with defined response protocols (target: 6); Time from alert to coordinated response activation (target: < 4 hours); Medical countermeasure stockpile readiness (target: 90%+) | Response exercise reports; Protocol documents; MCM inventory audits | Response protocols are approved by all ministries; MCM procurement is funded; Regular exercises are conducted |
| **Output 5** | Sustainable biodefense workforce and governance framework | Number of trained biodefense professionals (target: 200+ in 5 years); Regulatory framework covering biosafety, biosecurity, dual-use (target: comprehensive legislation); International partnerships active (target: WHO, OIE, NATO CBRN COE +) | Training program records; Legislative gazette; Partnership MOUs | Training programs are funded; Legislative process is completed; International partners engage |
| **Activities** | See Section 7 (Program Stages) for detailed activity breakdown | Budget, timeline, and resource requirements per activity | Program management reports; Financial records; Sprint reviews | Budget is approved; Staff are hired; Technology procurement proceeds on schedule |

  Train staff        Surveillance net   is operational      biodefense

## 4. LAYER 2 — MSP PROGRAM STRUCTURE

### 4.1 MSP Governance Model

```
┌─────────────────────────────────────────┐
│         SPONSORING GROUP                │  National decision-maker / Minister
│         (Strategic direction)           │  Approves program, allocates budget
├─────────────────────────────────────────┤
│         PROGRAM BOARD                   │  Cross-ministry steering committee
│         (Oversight & decisions)         │  Reviews progress, resolves conflicts
├─────────────────────────────────────────┤
│         PROGRAM DIRECTOR                │  Dr. Majed
│         (Day-to-day leadership)         │  Manages delivery, reports to Board
├─────────┬───────────┬───────────────────┤
│ Project │ Project   │ Project           │  Individual project managers
│ Manager │ Manager   │ Manager           │  Deliver specific outputs
│ (Hub)   │ (LIMS)    │ (GeoIntel)        │
└─────────┴───────────┴───────────────────┘
```

### 4.2 MSP 7 Principles Applied to BioR

| MSP Principle | Application to BioR |
|:---|:---|
| **1. Remaining aligned with corporate strategy** | Program aligns to Vision 2030 health security goals and WHO IHR obligations |
| **2. Leading change** | Program Director leads transformation across fragmented ministries toward integration |
| **3. Envisioning and communicating a better future** | Vision Statement and Blueprint describe Saudi Arabia with full biodefense capability |
| **4. Focusing on benefits and threats to them** | Benefits Map tracks 5 outputs → measurable national security improvements |
| **5. Adding value** | Each tranche delivers usable capability — not just documents |
| **6. Designing and delivering a coherent capability** | 5 Core Functions (Detect, Characterize, Assess, Respond, Sustain) form one integrated system |
| **7. Learning from experience** | Sprint Reviews, retrospectives, and international benchmarking drive continuous improvement |

### 4.3 MSP Blueprint (Summary)

The Blueprint describes **what Saudi Arabia's biodefense capability looks like when the program is complete**:

| Dimension | Current State | Future State (Blueprint) |
|:---|:---|:---|
| **Organization** | Fragmented across 6+ ministries | Unified coordination under one program authority |
| **Processes** | Ad hoc, event-driven response | Continuous detect-characterize-assess-respond cycle |
| **Technology** | Isolated systems, no integration | Integrated national platform (BioR) connecting all data sources |
| **Information** | Siloed, delayed, incomplete | Real-time, multi-sector, actionable intelligence |
| **People** | Scattered expertise, no dedicated biodefense workforce | 200+ trained professionals in dedicated biodefense roles |
| **Culture** | Reactive — respond after events | Proactive — continuous monitoring and preparedness |

### 4.4 MSP Tranches (Delivery Phases)

The program is delivered in **4 Tranches**, each ending with a review and go/no-go decision:

```
TRANCHE 1          TRANCHE 2          TRANCHE 3          TRANCHE 4
Foundation         Core Capability    Full Integration   Maturity
(0-12 months)      (12-30 months)     (30-48 months)     (48-60+ months)
     │                  │                  │                  │
  Approval           Detection          Response          Permanent
  Team setup         Lab network        Multi-sector      Institution
  Proof of concept   Assessment         International     Continuous
  Pilot sites        Pilot response     Full coverage     improvement
     │                  │                  │                  │
  REVIEW 1           REVIEW 2           REVIEW 3          REVIEW 4
  (Go/No-Go)        (Go/No-Go)         (Go/No-Go)        (Transition)
```

| Tranche | Duration | Focus | Key Deliverables | Gate Decision |
|:---|:---|:---|:---|:---|
| **T1 — Foundation** | Months 0–12 | Establish the program, prove concept, build team | Program charter approved; Core team hired; Technology platform (BioR) validated; 3 pilot sites connected; Baseline assessment complete | "Should we continue and expand?" |
| **T2 — Core Capability** | Months 12–30 | Build the 5 core functions to operational status | National LIMS operational (20+ labs); Surveillance network active; Threat assessment capability running; Response protocols drafted; Training program launched | "Are we achieving the intended benefits?" |
| **T3 — Full Integration** | Months 30–48 | Connect all sectors, achieve national coverage | All 6 sectors integrated; National coverage (100%); International partnerships active; Response exercises conducted; Regulatory framework submitted | "Are we ready for permanent operations?" |
| **T4 — Maturity** | Months 48–60+ | Transition to permanent institution | Legislative mandate; Permanent budget line; Full workforce; Continuous improvement cycle; International evaluation (WHO JEE) | "Transition from program to institution" |

## 7. LAYER 5 — PROJECT EXECUTION (PMI / AGILE HYBRID)

### 7.1 Execution Methodology

| Aspect | Method | Rationale |
|:---|:---|:---|
| **Technology development** | Agile/Scrum (2-week sprints) | Software requires iteration, feedback, and flexibility |
| **Program proposal & governance documents** | Waterfall (sequential) | Documents must flow logically: vision → scope → budget → approval |
| **Stakeholder engagement** | PMI Stakeholder Management | Structured identification, analysis, and engagement planning |
| **Risk management** | PMI Risk Management | Risk register, probability/impact matrix, mitigation plans |
| **Benefits tracking** | MSP Benefits Management | Map benefits to outputs, measure continuously |

### 7.2 Sprint Structure (for Technology Projects)

| Element | Detail |
|:---|:---|
| **Sprint length** | 2 weeks |
| **Sprint planning** | First day — define scope and stories |
| **Daily standup** | 15-minute status (when team exists) |
| **Sprint review** | Demo deliverables at end of sprint |
| **Sprint retrospective** | What went well, what to improve |
| **Definition of done** | Code deployed, tested, documented, committed |

### 7.3 Document Workflow (for Proposal Documents)

```
DRAFT → INTERNAL REVIEW → REVISION → PROGRAM DIRECTOR APPROVAL → FINAL
```

Each document follows a sequential flow. No document at Level 2 is started before its Level 1 prerequisite is complete.

## 10. QUALITY STANDARDS

### 10.1 Document Quality Criteria

Every document in this program must meet:

| Criterion | Standard |
|:---|:---|
| **Evidence-based** | Claims supported by data, references, or established frameworks |
| **Audience-appropriate** | Written for the intended reader, not the author |
| **Internally consistent** | No contradictions with other program documents |
| **Measurable** | Where possible, include quantifiable indicators |
| **Actionable** | Each document leads to a clear next step |
| **Version-controlled** | All documents in Git with change history |

### 10.2 Review Process

```
DRAFT → SELF-REVIEW → REVISION → APPROVED → COMMITTED TO GIT
```
